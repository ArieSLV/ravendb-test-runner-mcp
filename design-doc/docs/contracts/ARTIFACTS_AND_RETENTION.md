# Artifacts and Retention Contract

## Purpose

Define artifact kinds, storage routing, retention classes, cleanup behavior, and recovery rules.

## Canonical storage principle

- Metadata and searchable summaries live in RavenDB Embedded.
- Large raw artifacts live on filesystem.
- Compact artifacts may be duplicated or stored as attachments when policy allows.

## Artifact taxonomy

### Mandatory artifact kinds

- `plan`
- `command`
- `env`
- `console`
- `stdout`
- `stderr`
- `trx`
- `junit`
- `diag`
- `blame`
- `normalized`
- `repro`
- `compare`

### Artifact ownership

- `plan`, `command`, `env`, `normalized`, `repro`, `compare` are generated by the control plane
- `stdout`, `stderr`, `console`, `trx`, `junit`, `diag`, `blame` originate from or around execution

## Filesystem layout

```text
<artifact-root>/
  workspaces/<workspace-id>/
    runs/<run-id>/
      plan/
      step-001/
        console/
        results/
        diagnostics/
      step-002/
        ...
      attempts/
        attempt-001/
          step-001/
          ...
        attempt-002/
          ...
      compare/
      normalized/
```

## Naming conventions

- file names MUST be deterministic
- step indexes MUST be zero-padded where used in names
- attempt indexes MUST be zero-padded where used in names
- sensitive artifacts MUST be flagged in metadata

## Retention classes

RetentionClass enum:
- `ephemeral`
- `standard`
- `important`
- `audit`
- `manual-hold`

### Default mapping

- `stdout` / `stderr` / `console` => `standard`
- `trx` / `junit` => `important`
- `diag` / `blame` => `important`
- `normalized` / `repro` / `compare` => `audit`
- temporary fragments => `ephemeral`

## Cleanup rules

### Ephemeral
- safe to purge earliest
- may be purged automatically after successful normalization

### Standard
- retained according to global default retention window

### Important
- retained longer than standard
- especially when the run failed or timed out

### Audit
- retained with highest default priority unless manually overridden

### Manual hold
- never auto-delete until hold is removed

## Attachment routing rules

If artifact size <= compact threshold:
- MAY be stored as a RavenDB attachment
- MUST still have a `RunArtifact` metadata document

If artifact size > compact threshold:
- MUST be filesystem-backed
- MAY have a compact preview or summary stored in RavenDB

## Preview generation

The system SHOULD generate previews for:
- text logs
- normalized result summaries
- compare outputs
- skip/failure explanation bundles

Previews MUST be redacted according to `SECURITY_AND_REDACTION.md`.

## Rebuild and recovery

If filesystem artifacts are present but metadata is missing:
- mark as orphaned artifact
- do not silently delete
- allow recovery tooling

If metadata is present but filesystem payload is missing:
- mark artifact as missing
- preserve metadata
- surface consistency warning in UI and diagnostics

## Validation requirements

- artifact routing tests
- retention classification tests
- preview generation tests
- orphan detection tests
- missing payload recovery tests
- cleanup journal tests
